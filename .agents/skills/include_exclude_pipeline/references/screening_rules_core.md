### TIER 0: Fast-Exit Filters

**Screening Rule 7: Data Extractability Gate**
- Qualitative studies (interviews, case studies, grounded theory, etc.) that do not provide extractable quantitative effect sizes ($r$) or formal correlation matrices MUST be EXCLUDED. Assign Exclusion Code 1 (No effect size of interest), as they do not provide the statistical data required for this meta-analysis.
- Do not be fooled by tables filled with numbers. If boundary spanning is used solely as a moderator to split a sample (e.g., Spanners vs. Non-spanners), or if the reported correlations are between boundary spanning and *regression residuals* (predictive errors) rather than substantive variables, you CANNOT extract these as zero-order effect sizes. The paper must be excluded under Code 1.
- If a paper lacks a formal correlation matrix but reports the $R^2$ value for a *simple* linear regression (one predictor, zero control variables), you MUST extract the zero-order correlation using the formula $r = \sqrt{R^2}$. Do not erroneously exclude the paper for lacking a matrix if this data is present.
- ⚠️ WARNING (SEM-only Data Warning): If a paper relies entirely on SEM path coefficients and does NOT provide a zero-order correlation matrix (even latent), it must be excluded for lacking extractable effect sizes. Do NOT confuse partial rectangular cross-correlation tables with full square correlation matrices.

### TIER 1: Override Gates (반드시 먼저 확인 — CHECK FIRST)

**Screening Rule 1: Leader BSB & Cross-Entity Inclusion (OVERRIDE PRIORITY)**
The concept of Boundary Spanning Behavior is officially expanded to include **Boundary Spanning Leadership (BSL)**. If a study measures the boundary spanning behavior of a leader/manager as rated by their subordinates (cross-entity dyadic data), it MUST BE INCLUDED. Do not exclude a paper simply because the respondent is rating their supervisor's behavior rather than their own. **MANDATORY SEQUENCING:** Before applying ANY exclusion rule (Code 1, Code 3, Screening Rule 6, etc.), the screening agent MUST first check whether Screening Rule 1 or Screening Rule 2 applies. If the paper qualifies under Screening Rule 1 or 2, it is INCLUDED and no further exclusion logic may override this verdict. **CRITICAL OVERRIDE:** This rule takes absolute precedence over any "Cross-Entity Trap" or Code 1 exclusion logic. Leader BSB IS BSB — the construct is valid regardless of who the rater is. Code 1 applies ONLY when the construct itself is not BSB (e.g., Work-Family boundary management, Adaptive Selling). **Leader BSB Trap Warning:** Do not be fooled by the label "Boundary-spanning leadership". If the items measure internal demographic diversity management (e.g., bridging age gaps within a single branch using Blau's index), it is inclusive leadership, not structural BSB (Exclude under Code 1 & 3). **N=Leaders=N=Teams Exception:** When exactly one leader exists per team, the effective N for BSB is N_leaders (individual-level), NOT N_teams (team-level). Do not apply Code 3 to such designs.

**Screening Rule 2: Intra-Organizational BSB Inclusion (OVERRIDE PRIORITY)**
Boundary spanning behavior across internal organizational boundaries (e.g., cross-functional teamwork, cross-departmental knowledge sharing between IT and business units, advice-seeking networks across departmental boundaries) MUST BE INCLUDED. Do not exclude a paper simply because the focal employee interacts with other internal departments rather than external organizations. **CRITICAL OVERRIDE:** Intra-organizational BSB IS BSB — the construct is valid regardless of boundary direction (internal vs. external). Code 1 applies ONLY when the construct itself is not BSB. **Network Degree Centrality Exception:** Ego-network measures (e.g., degree centrality of advice-seeking ties across unit boundaries, E-I index of cross-boundary contacts) represent purposive intra-organizational boundary spanning and MUST BE INCLUDED. Do not confuse structural network BSB with mere communication frequency.
📌 NOTE (Implanted Boundary Spanner Trap): If a study's sample consists of employees whose job role is boundary spanning (e.g., logistics implants on-site at client facilities, expatriates) AND the study measures their knowledge exchange, coordination, or information-sharing behavior with the partner organization, this IS a valid BSB construct → INCLUDE. Do NOT exclude just because the paper labels it "knowledge exchange" instead of "boundary spanning behavior".

**Screening Rule 3: Zero False Negative & Independent Verification (Safety Net)**
Never blindly execute bulk inclusion or exclusion lists without independent methodological verification. When reviewing exclusion candidates or candidate lists:
   - **Beware of Title Keywords:** Words like *"Interorganizational"*, *"Selling Organizations"*, or *"Modelling"* in titles DO NOT automatically mean Macro (Code 3) or Conceptual (Code 4).
   - **Mandatory Sample Verification:** You must explicitly open and check the abstract/methodology to verify whether the study collected quantitative survey data from individual organizational employees (e.g., $N=225$ sales representatives, frontline staff, bank managers).
   - **Protect Pristine Empirical Studies:** If a study reports individual employee boundary spanning behavior and statistical effect sizes ($r$), it MUST BE SAVED AND INCLUDED (`1 = Include`), overriding any surface keyword heuristics.

### TIER 2: Exclusion Filters (Tier 1을 통과한 경우에만 적용)

**Screening Rule 4: Construct Homonymy & Level-of-Analysis Guardrail**
Never rely on keyword presence ("Face Validity") when screening for Boundary Spanning Behavior (BSB). The presence of the word "boundary-spanning" does not guarantee inclusion. BSB must be an interpersonal behavior performed by an individual employee (e.g., frontline worker, salesperson, leader). Due to construct homonymy across disciplines, agents must explicitly verify the **Unit of Analysis ($N$)** and **Entity Type**:
   - **CEO vs. Entrepreneur Guardrail (Population Boundaries):** Top Management Team (TMT) members and CEOs MUST be INCLUDED as they are high-ranking employees performing inter- or intra-organizational boundary spanning. However, independent Entrepreneurs, founders, and self-employed individuals MUST be EXCLUDED (Code 2). Entrepreneurs operate independently rather than as embedded employees, and studies examining their "boundary management" almost universally measure Work-Life boundaries (Code 1) or firm-level strategies (Code 3), not interpersonal BSB.
   - **Firm/Patent/Alliance Level (Code 3):** Studies analyzing patent citations, interfirm alliances, corporate R&D networks, firm financial valuation (e.g., SMJ, JIBS patent/alliance papers), or strategic alliances at the Firm-level (e.g., MNEs) MUST be excluded under `Code 3`.
   - **Transaction / Buying Situation Level (Code 3):** Studies adopting specific transactions, buying situations, or buyer-supplier relations as the unit of analysis, particularly those using multi-rater "consensus" to evaluate organizational relationships, MUST be excluded under `Code 3`.
   - **Dyadic Data Guardrail (Code 3):** If a study analyzes data at the "dyad" or "relationship" level (e.g., $N=673$ ties reported by 87 individuals), it MUST be excluded under `Code 3`. The correlation matrix from such studies violates statistical independence (repeating individuals across multiple observations) and yields no valid effect sizes. **CRITICAL:** Watch out for PLS-SEM path models using degree-symmetric dyadic consensus variables (e.g., measuring relationship quality where N = relationships). These still violate statistical independence and must be excluded. ⚠️ EXCEPTION: Cross-entity leader-subordinate ratings where each subordinate rates exactly one leader are NOT dyadic data — see Override Gate: Screening Rule 1 (Tier 1). 📌 NOTE (1:1 Matched Dyad Safe Harbor): If a study uses 1-to-1 matched pairs (e.g., 1 expatriate paired with exactly 1 coworker) and each individual appears only once in the dataset, statistical independence IS preserved. Do NOT apply Code 3. Only apply Code 3 when N = relationships (e.g., 87 individuals generating 673 dyadic ties). 📌 NOTE (AI Tool Interaction): BSB must be an interpersonal, human-to-human behavior. If the "boundary spanning" measured involves using Generative AI tools (e.g., ChatGPT) to gather information, this is human-computer interaction and lacks BSB construct validity → Exclude under Code 1.
   - **Team Level Aggregation (Code 3):** Studies reporting team as the unit of analysis ($N = \text{teams}$, e.g., 82 NPD teams) where correlations represent team-level aggregated constructs MUST be excluded under `Code 3`. **WARNING:** This includes Group-level Diversity Indices (e.g., calculating Generational Diversity using Blau's index based on branch composition) even if the individual respondent is mentioned.
   - **Work-Family Boundary Management (Code 1):** If the analysis involves Work-Family or Work-Life boundary management rather than inter- or intra-organizational boundary spanning, it MUST be EXCLUDED under `Code 1`.
   - **Non-Employee / Bibliometrics / Students (Code 2):** Studies using citation databases (Web of Science), student samples (e.g., MBA students), consumers, or general citizens without organizational employee data MUST be excluded under `Code 2`.

**Screening Rule 5: Proxy Guardrail (Firm-Level Informants & Survey Referents)**
Watch out for studies where individuals (e.g., Managers, CEOs, or Sales Representatives) are surveyed merely as key informants or proxies to measure Firm-level or Team-level boundary spanning (e.g., inter-firm alliances, organizational networks, B2B integration). Even if the subject is an individual "manager", if the construct represents the "firm's external relationships", it MUST be EXCLUDED under the Firm/Team Level Ban. **CRITICAL:** Always check the grammatical subject (referent) of the survey items. If the items ask about "We", "Our team", or "My organization" rather than "I", the respondent is acting as a macro-level Key Informant, and the study MUST be EXCLUDED (Code 3). **CEO vs Firm Exception:** If a CEO rates their *own* individual behavior (e.g., "I absorb outside pressures", N=firms=N=CEOs), this is valid individual-level BSB. Do not confuse it with firm-level alliances.

**Screening Rule 6: Purposive Action vs. Mere Communication (incl. Adaptive Selling)**
Boundary Spanning Behavior (BSB) requires intentional, purposive actions (e.g., seeking resources, scanning for information, coordinating tasks). Variables that merely measure "communication frequency" (e.g., how often someone emails or chats) or *percentage of time* spent with outsiders without specifying a purposive boundary-spanning action are INVALID and MUST be excluded under Code 1. **VALID Network BSB (Do NOT Exclude):** The following network-based measures ARE purposive and MUST BE INCLUDED: (a) Advice-seeking network degree centrality (asking for work-related advice across boundaries), (b) E-I index measuring cross-boundary contacts for knowledge exchange, (c) Ego-network heterogeneity based on purposive boundary-spanning ties (e.g., seeking expertise from other units). **INVALID (Exclude):** (a) Pure email/phone/chat frequency counts without purposive content, (b) Percentage of time spent with outsiders without specifying action type, (c) Sociometric sensor data measuring mere proximity or speaking events, (d) Burt's structural constraint measures.
📌 NOTE (Adaptive Selling Exception): Do not confuse sales tactics or personality traits (like Flexibility or Adaptive Selling, e.g., the Spiro & Weitz ADAPTS scale) with Boundary Spanning Behavior. Adjusting one's approach to suit a customer is a distinct construct from active information scouting, representation, or task coordination across boundaries. Studies measuring only adaptive selling lack an effect size of interest (Code 1).
📌 NOTE (Test Case): Cross-departmental communication frequency (e.g., how often employees email members of other departments) without specified purposive boundary-spanning action → INVALID (Code 1). This is mere communication, not purposive BSB.

### TIER 3: Post-Decision Coding & Documentation

**Screening Rule 8: Exclusion Coding Protocol**
When excluding a paper, use the standardized exclusion code in the `Reason for Exclusion` column (Col 6) and put the detailed explanation in the `Notes` column (Col 16). Available codes: `1 = No effect size of interest`, `2 = Non-employee samples`, `3 = Non-individual level (team/firm/org analysis)`, `4 = Non-primary study`, `5 = Multiple reasons (specify in Notes)`, `6 = Duplicate`, `7 = Non-English language`, `99 = Other (specify in Notes)`.
   - **Code 1 Unification:** If a study measures the wrong construct entirely (e.g., Work-Family boundary management instead of inter-organizational or intra-organizational boundary spanning), unify the exclusion under Code 1, as it lacks the target effect size.
   - **Code 5 Traceability:** If a study simultaneously violates multiple fundamental inclusion criteria (e.g., macroeconomic data lacking both individual analysis and human samples), use Code 5 and explicitly specify the sub-codes in the Notes column (e.g., `[Code 2 & Code 3]`) for perfect PRISMA flow traceability.
   ⚠️ WARNING (Multi-Study Trap): When a paper contains multiple independent studies or chapters, each part may violate a different guardrail (e.g., firm-level patent analysis in Study 1 vs. student samples in Study 2). Instead of picking just one reason, you MUST capture both flaws using Code 5 (e.g., `[Code 2 & Code 3]`).
