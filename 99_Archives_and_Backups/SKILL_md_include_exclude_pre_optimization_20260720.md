# Skill: Include/Exclude Pipeline Orchestrator

When the user asks you to "run the pipeline" or "파이프라인 돌려", you must act as the Master Orchestrator for the Include/Exclude screening process using Real Subagents. Do NOT use the dummy `run_batch.py` script.

## 1. Batch Payload Generation
- Run `python .agents/skills/include_exclude_pipeline/scripts/swarm_prep.py --batch 40`
- This script checks the target Excel file and generates a payload file at `scratch/subagents_payload.json` containing up to 40 pending papers.
- If the script outputs that no more pending papers are found, proceed to Step 4 (Reporting).

## 2. Real-Swarm Invocation
- Read the generated `scratch/subagents_payload.json` file.
- Use your `invoke_subagent` tool to spawn the `bsma_reviewer_v3` subagents exactly as defined in the JSON array.
- **Wait** asynchronously for all subagents to finish processing their PDFs and output their verdict JSONs into `scratch/outputs/`.

## 3. Data Injection & Loop
- Once the subagents finish, run `python .agents/skills/include_exclude_pipeline/scripts/swarm_inject.py` to write their verdicts into the Excel sheet and clean up the JSON files.
- After injection completes, **loop back to Step 1** to process the next batch of 40 papers until the Excel queue is completely empty.

## 4. Final Reporting
- Once the Excel queue is completely processed, intermediate batch reports will have been automatically generated by `swarm_inject.py`.
- To generate a comprehensive final Match Rate Report across all 701 papers, adapt and run `python .agents/skills/include_exclude_pipeline/scripts/grade_test.py` or write a custom cross-validation script.

The presence of the word "boundary-spanning" does not guarantee inclusion. BSB must be an interpersonal behavior performed by an individual employee (e.g., frontline worker, salesperson, leader). Due to construct homonymy across disciplines, agents must explicitly verify the **Unit of Analysis ($N$)** and **Entity Type**:
   - **CEO vs. Entrepreneur Guardrail (Population Boundaries):** Top Management Team (TMT) members and CEOs MUST be INCLUDED as they are high-ranking employees performing inter- or intra-organizational boundary spanning. However, independent Entrepreneurs, founders, and self-employed individuals MUST be EXCLUDED (Code 2). Entrepreneurs operate independently rather than as embedded employees, and studies examining their "boundary management" almost universally measure Work-Life boundaries (Code 1) or firm-level strategies (Code 3), not interpersonal BSB.
   - **Firm/Patent/Alliance Level (Code 3):** Studies analyzing patent citations, interfirm alliances, corporate R&D networks, firm financial valuation (e.g., SMJ, JIBS patent/alliance papers), or strategic alliances at the Firm-level (e.g., MNEs) MUST be excluded under `Code 3`.
   - **Transaction / Buying Situation Level (Code 3):** Studies adopting specific transactions, buying situations, or buyer-supplier relations as the unit of analysis, particularly those using multi-rater "consensus" to evaluate organizational relationships, MUST be excluded under `Code 3`.
   - **Dyadic Data Guardrail (Code 3):** If a study analyzes data at the "dyad" or "relationship" level (e.g., $N=673$ ties reported by 87 individuals), it MUST be excluded under `Code 3`. The correlation matrix from such studies violates statistical independence (repeating individuals across multiple observations) and yields no valid effect sizes. **CRITICAL:** Watch out for PLS-SEM path models using degree-symmetric dyadic consensus variables (e.g., measuring relationship quality where N = relationships). These still violate statistical independence and must be excluded.
   - **Team Level Aggregation (Code 3):** Studies reporting team as the unit of analysis ($N = \text{teams}$, e.g., 82 NPD teams) where correlations represent team-level aggregated constructs MUST be excluded under `Code 3`. **WARNING:** This includes Group-level Diversity Indices (e.g., calculating Generational Diversity using Blau's index based on branch composition) even if the individual respondent is mentioned.
   - **Work-Family Boundary Management (Code 1):** If the analysis involves Work-Family or Work-Life boundary management rather than inter- or intra-organizational boundary spanning, it MUST be EXCLUDED under `Code 1`.
   - **Non-Employee / Bibliometrics / Students (Code 2):** Studies using citation databases (Web of Science), student samples (e.g., MBA students), consumers, or general citizens without organizational employee data MUST be excluded under `Code 2`.

**Screening Rule 8: Qualitative Study Exclusion (Strict Correlation Requirement):** Qualitative studies (interviews, case studies, grounded theory, etc.) that do not provide extractable quantitative effect sizes ($r$) or formal correlation matrices MUST be EXCLUDED. Assign Exclusion Code 1 (No effect size of interest), as they do not provide the statistical data required for this meta-analysis.

**Screening Rule 9: Illusion of Data Guardrail (Moderators & Residuals):** Do not be fooled by tables filled with numbers. If boundary spanning is used solely as a moderator to split a sample (e.g., Spanners vs. Non-spanners), or if the reported correlations are between boundary spanning and *regression residuals* (predictive errors) rather than substantive variables, you CANNOT extract these as zero-order effect sizes. The paper must be excluded under Code 1.

**Screening Rule 10: R-squared Extraction (Simple Regressions):** If a paper lacks a formal correlation matrix but reports the $R^2$ value for a *simple* linear regression (one predictor, zero control variables), you MUST extract the zero-order correlation using the formula $r = \sqrt{R^2}$. Do not erroneously exclude the paper for lacking a matrix if this data is present.

- **1:1 Matched Dyad Safe Harbor (Screening Rule 1 supplement):** If a study uses 1-to-1 matched pairs (e.g., 1 expatriate paired with exactly 1 coworker) and each individual appears only once in the dataset, statistical independence IS preserved. - **AI Tool Interaction Exclusion (Screening Rule 1 supplement):** BSB must be an interpersonal, human-to-human behavior. - **SEM-only Data Warning (Extraction Rule 1 supplement):** If a paper relies entirely on SEM path coefficients and does NOT provide a zero-order correlation matrix (even latent), it must be excluded for lacking extractable effect sizes. - **Network-Based BSB Inclusion (Screening Rule 5+6 supplement):** If a study measures cross-boundary ties using a network generator that asks about purposive actions (e.g., "who do you go to for information or knowledge on work-related topics"), the resulting network variables (bridging ties, cross-boundary contacts, degree centrality) ARE valid BSB operationalizations → INCLUDE. - **Implanted Boundary Spanner / Knowledge Exchange (Screening Rule 5 supplement):** If a study's sample consists of employees whose job role is boundary spanning (e.g., logistics implants on-site at client facilities, expatriates) AND the study measures their knowledge exchange, coordination, or information-sharing behavior with the partner organization, this IS a valid BSB construct → INCLUDE. Do NOT exclude just because the paper labels it "knowledge exchange" instead of "boundary spanning behavior".
